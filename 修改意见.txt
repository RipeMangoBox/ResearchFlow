2. 阅读图

这是 Obsidian 里的图。

这一层必须极度克制。
我建议用户可见节点只保留 4 类：

二、现在最该改的，不是“图谱算法”，而是 Obsidian 的导出对象

你现在的导出层太接近数据库结构了。

我建议用户可见层只导出 5 种笔记
1. 方向总览

它不应该成为所有论文都回链的超级节点。

2. Paper Note

建议分层：

3. Concept Note

这个不是从单篇论文直接落地的。

你说得对：

根因是什么
现在主要解法有哪些
哪些只是 patch
哪些更有希望
5. Lineage Note

这类文档非常重要。

不是让 Graph 自动替你显示脉络，而是明确写一条主线，例如：

DPO family
Reward design line

Lineage Note 是“人类可读的发展脉络图”，不是数据库里 delta_card_lineage 的直接 dump。

七、为什么你现在会出现“大量单节点”

你自己已经观察到了，本质原因是三个：

1. 14 篇论文不够形成“骨架”

不是数量太少，而是样本结构不完整。

一个方向包至少要有：

3–5 篇 baseline anchors
5–8 篇 structural papers
5–8 篇 plugin papers
3–5 篇 adjacent transfer papers

如果只选了 14 篇，而且里面缺：

canonical baseline
主线分叉
典型 plugin 对照
相邻领域迁移样本

那图天然会断。

2. 你的导出规则没有“最低连边契约”

每篇论文如果只是一个文件，没有强制连到：

baseline
concept
bottleneck
lineage
same-family papers

它就必然是孤点。

3. 你用错了图的入口

Graph view 只显示内部链接和局部/全局连接，适合看局部关系，不适合直接承担“研究导航”。Obsidian 也明确把 Local Graph 作为从当前 note 向外看若干层连接的方式。

八、重新设计链接策略：把“用户可见链接”做瘦

这是你现在最该立刻改的。

核心原则

数据库里可以有很多边，导出到 Obsidian 的可见链接必须少而强。

我建议每篇主论文导出时，正文里的 [[wikilinks]] 最多只保留：

2–3 个 concepts

00_Home/*

九、服务器端的 agent 也要重设计：不是一个“论文分析 agent”就够了

你说得对：
现在的“论文提取 agent”大概率太简单了。

被论文故事牵着走
只对比论文自己提到的 baseline
第二篇进来后不会反过来更新第一篇的对比
concept / bottleneck / lineage 生成都是单向的
正确做法是拆成专业角色

Claude Code 官方支持 subagents，它们可以在独立上下文里运行、隔离高体量搜索或日志，并支持并行研究；hooks 则适合做确定性动作。你的服务器端不需要长期跑一个 Claude Code 会话，但非常适合按这个思路设计“多角色工作流”。

十、服务器端如何让 API 调用更稳定

你问的是“如何让 API 调用更稳定，如何优化”。
我的建议是不要继续增加“大接口”，而是把一次分析拆成多个小而稳定的任务。

1. 不要让一个接口跑完整智能链

现在 run_full_pipeline 这种入口适合作为 orchestrator，但内部不能是单次长调用。

应该拆成：

collect_candidates
resolve_assets
extract_evidence
build_delta_card
build_compare_set
propose_lineage
synthesize_concept
synthesize_bottleneck
render_exports

每步单独：

有 schema
有重试
有中间产物
有状态
2. 所有 LLM 输出都做严格 schema 验证

不要接受“大 JSON 大而全”。

每个阶段只输出自己应该负责的字段。
例如：

extract_evidence

只输出：

key_equations[]
key_figures[]
key_tables[]
evidence_spans[]
build_delta_card

只输出：

baseline_papers
changed_slots
unchanged_slots
mechanisms
paper_bottleneck_claims
same_family_candidates
lineage_judge

只输出：

parent_candidates
relation_type
confidence
rationale
evidence_refs

这样失败更好重试，也更稳定。

3. 做 idempotency 和版本化

每次分析至少带：

paper_id
source_asset_hash
analysis_schema_version
model_version
prompt_version

这样：

同一资产不会重复跑
模型升级时能重算
回滚有依据
4. 对外部源做缓存和 checkpoint

对：

OpenAlex
Semantic Scholar
GitHub
Zotero

都做：

source_sync_runs
incremental_checkpoints
ETag/hash
retry/backoff

Semantic Scholar 本身就把 Academic Graph 和 Recommendations 分成独立 API；这很适合你把“检索元数据”和“邻域扩展”拆开。

5. 做 provider fallback

不要让一个模型失败，整条链就挂。

建议：

强阅读：Claude
