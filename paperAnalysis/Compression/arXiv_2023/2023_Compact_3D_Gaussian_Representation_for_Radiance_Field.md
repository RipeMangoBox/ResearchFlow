# Compact 3D Gaussian Representation for Radiance Field

> [!abstract] **Quick Links & TL;DR**
> - **Links**: [arXiv](https://arxiv.org/abs/2311.13681), [Project](https://maincold2.github.io/c3dgs/)
> - **Summary**: 这篇论文把 3DGS 的压缩拆成“删掉不重要的高斯 + 共享高斯属性表示”两步，在基本保持重建质量的同时大幅降低存储并提升渲染速度。
> - **Key Performance**: Deep Blending 上达到 **29.79 PSNR / 180.6 FPS / 43.2 MB**；加入简单后处理后，平均可实现 **25×+** 压缩

> [!info] **Agent Summary**
> - **task_path**: 多视角有位姿图像 / 静态场景 -> 紧凑 3D Gaussian radiance field -> 实时新视角渲染
> - **bottleneck**: 3DGS 为了保真度会 densify 出大量冗余高斯，且逐高斯存储 SH 颜色与几何属性，导致存储、显存和渲染开销线性膨胀
> - **mechanism_delta**: 用体积+透明度联合可学习掩码持续删减低贡献高斯，再用哈希网格神经场共享颜色、用 R-VQ 共享尺度与旋转
> - **evidence_signal**: Deep Blending 上在 43.21 MB 下达到 29.79 PSNR / 180.60 FPS，优于 3DGS* 的 29.46 PSNR / 132.47 FPS / 663.32 MB
> - **reusable_ops**: [体积与透明度联合结构剪枝, 哈希网格共享视角相关属性, 残差向量量化代码本, 测试前预计算共享特征]
> - **failure_modes**: [极低码率下几何代码本容量不足会明显掉点, 部分真实场景上颜色神经场替代 SH 会带来轻微画质回退]
> - **open_questions**: [动态场景能否复用同一压缩策略, 颜色神经场如何进一步量化而不损失实时性]

## Part I：问题与挑战

这篇论文解决的不是“3DGS 能不能渲染得快”，而是 **3DGS 已经够快之后，是否还能足够小、足够省内存、足够易部署**。

### 真正的问题是什么
3D Gaussian Splatting 相比 NeRF 最大的优势是把体渲染换成了栅格化，因此渲染速度非常快；但它把瓶颈从“每条光线要采很多点”转移成了：

1. **高斯数量爆炸**  
   训练中的 densification（clone / split）会不断增加高斯数量，以保住细节。
2. **逐高斯属性过重**  
   每个高斯都要存位置、透明度、尺度、旋转，以及高维 SH 颜色参数。
3. **渲染复杂度近似随高斯数线性增长**  
   所以冗余高斯不仅占存储，也直接拖慢渲染。

换句话说，3DGS 的核心矛盾不是“表示能力不够”，而是 **为了高质量表示，付出了过高的表示冗余**。

### 为什么现在值得解决
因为 3DGS 已经把“实时渲染”打开了，但若单个场景经常需要数百 MB 甚至 1GB+ 存储，那么它仍然很难走向：

- 设备侧部署
- 大规模场景库管理
- 低显存/低带宽环境
- 快速加载与分发

所以这篇论文的时机点很明确：**先有了 3DGS 的速度基础，现在该解决它的体积问题。**

### 输入 / 输出接口
- **输入**：带相机位姿的多视角图像 + SfM/COLMAP 稀疏点初始化
- **输出**：一个场景级、可实时渲染的紧凑 3D Gaussian 表示
- **目标任务**：静态场景的新视角合成 / 3D 场景表示

### 边界条件
这篇方法默认：

- 是 **静态场景**
- 依赖 **SfM 初始化**
- 仍建立在 **3DGS 渲染管线** 上
- 是 **场景级优化**，不是通用前馈模型

## Part II：方法与洞察

作者的设计非常清楚：**不改 3DGS 的渲染范式，只压缩 3DGS 的表示熵。**

它从两个轴同时下手：

- **数量压缩**：减少活跃高斯数
- **属性压缩**：让剩余高斯不再各自独占一套完整属性

### 方法拆解

#### 1. 可学习 Gaussian Volume Mask：先删“几乎没贡献”的高斯
作者观察到，原始 3DGS 虽然会基于 opacity 做一些控制，但训练后仍有大量冗余高斯存在。  
他们增加了一个 **可学习二值掩码**，并且不是只看透明度，而是同时看：

- **opacity**
- **scale / volume**

关键点在于：**体积很小且透明度低的高斯，对最终像素贡献极弱**。  
因此它们是最适合被优先删除的对象。

更重要的是，这个 masking 不是训练后一次性做，而是：

- 在训练过程中持续学习
- 每次 densification 后都执行删除
- 一直保持较低的活跃高斯数

这改变了训练动态：不是先膨胀再冻结，而是 **边长边修剪**。

#### 2. 紧凑视角相关颜色：用共享神经场替代逐高斯 SH
原始 3DGS 中，视角相关颜色主要靠每个高斯自己的 SH 参数表示，这部分占参数大头。  
作者的替代思路是：

- 不再为每个高斯单独存完整颜色参数
- 用 **hash-grid + tiny MLP** 构成的神经场
- 输入高斯位置和视角方向
- 输出该高斯在当前视角下的颜色

本质上，这是把颜色从“逐点独立存储”改成“**空间共享函数**”。

为什么成立？因为相邻高斯的颜色/外观往往有强相关性，特别是在静态场景中，直接逐点存 SH 是一种高冗余设计。

#### 3. 几何代码本：用 R-VQ 共享尺度与旋转
作者进一步观察到，场景中的大量高斯虽然位置不同，但 **尺度和旋转模式并不高度多样**。  
因此他们把几何属性拆成：

- 小型 **代码本**
- 每个高斯只存 **索引**

具体采用的是 **Residual Vector Quantization (R-VQ)**：

- 第一层代码本匹配一个粗模式
- 后续层拟合残差
- 多阶段叠加近似原始尺度/旋转

这相当于把几何从“连续独立参数”变成“低熵字典组合”。

#### 4. 推理时预计算：把共享表示的代价提前消化
共享表示可能带来一个常见疑问：  
“既然你用了神经场和代码本，会不会把推理变慢？”

作者的答案是：**把能提前算的都提前算掉**。

- hash-grid 的视角无关部分可预计算
- R-VQ 的最近代码索引可预先确定

所以测试时真正的额外计算被压得很低，而高斯数量下降本身又直接加速 rasterization。

### 核心直觉

3DGS 的冗余不是单一来源，而是三种不同统计结构叠在一起：

1. **贡献长尾**：很多高斯几乎不影响结果  
2. **颜色局部相关**：邻近高斯没必要各自存完整视角相关颜色  
3. **几何低熵重复**：大量高斯的尺度/旋转模式相似

这篇论文真正改变的是：

> 从“每个高斯都是独立完整实体”  
> 变成  
> “只保留重要高斯 + 用共享结构表示剩余属性”。

于是瓶颈发生了因果转移：

- **What changed**：逐高斯独立存储 → 结构化删减 + 共享表示
- **Which bottleneck changed**：表示冗余和内存带宽压力下降
- **What capability changed**：在接近原始质量下得到更小模型和更快渲染

这也是它为什么有效：  
它不是盲目压缩，而是分别针对 **结构冗余、颜色冗余、几何冗余** 设计匹配的压缩器。

### 战略权衡表

| 设计选择 | 改变了什么瓶颈 | 收益 | 代价 / 风险 |
|---|---|---|---|
| 体积+透明度联合掩码 | 降低活跃高斯数 | 直接减少存储、显存和栅格化开销 | 过强掩码可能伤害细薄结构 |
| 哈希网格颜色场 | 去掉逐高斯高维 SH 存储 | 颜色参数显著共享，压缩比高 | 训练/渲染多出小 MLP；部分场景画质略退 |
| R-VQ 几何代码本 | 把连续几何变成索引化表示 | 几何存储进一步缩小 | 低码率时代码本表达力不足 |
| 测试前预计算 | 避免共享表示拖慢推理 | 保住实时性 | 实现更复杂，依赖离线预处理 |

### 一个关键实现判断
作者并没有把所有属性都硬塞进同一种表示里，而是“**按属性统计特性选压缩器**”：

- 颜色：适合神经场共享
- 几何：适合代码本量化
- 结构：适合掩码剪枝

这比“一种统一压缩方案压所有属性”更合理，附录里也显示 I-NGP 并不适合表示除颜色外的其它属性。

## Part III：证据与局限

### 关键证据信号

| 信号类型 | 观察 | 结论 |
|---|---|---|
| comparison | Mip-NeRF 360 平均从 **746.03 MB → 48.82 MB**，PSNR 仅 **27.46 → 27.08**，FPS **120.40 → 128.05** | 证明该方法的主收益是“几乎不伤质量的极大压缩” |
| comparison | Deep Blending 上 **29.79 PSNR / 180.60 FPS / 43.21 MB**，而 3DGS* 为 **29.46 / 132.47 / 663.32 MB** | 在真实场景上达到最强的质量-体积-速度折中 |
| comparison | NeRF-Synthetic 上 **33.33 PSNR, 5.55 MB, 545 FPS**，对比 3DGS 的 **33.32, 68.1 MB, 359 FPS** | 在合成场景上几乎零质量损失地实现 10×+ 压缩和 1.5× 加速 |
| ablation | 仅 masking 就能把 Playroom 的高斯数 **2.34M → 0.97M**，FPS **154 → 254**，PSNR 还略升 | 速度提升的核心因果来源是“减少活跃高斯” |
| analysis | 属性分解后，未做后处理时颜色表示仍占 **25.2 MB / 48.8 MB** | 下一步进一步压缩的重点仍是颜色分支 |

### 最值得记住的实验结论
这篇论文的能力跃迁，不是“绝对 PSNR 碾压 3DGS”，而是：

> **单位 MB、单位高斯数、单位渲染时间上的质量更高。**

也就是说，它把 3DGS 从“高质量但笨重”推向了“高质量且更可部署”。

### 关于训练效率的真实结论
要注意，**完整方法并不总比 3DGS 训练更快**。  
虽然 masking 本身能缩短训练时间，但引入颜色神经场和 R-VQ 后，整体训练通常会略慢于 3DGS*。因此这篇论文最硬的贡献是：

- 存储压缩
- 渲染加速
- 推理更省资源

而不是端到端训练时间全面领先。

### 局限性
- **Fails when**: 压缩被推到极低码率、尤其几何代码本容量过小或 R-VQ 阶段太少时，几何多样性无法被覆盖；另外在部分真实场景上，PSNR/LPIPS 相比 3DGS 仍会有小幅回退。
- **Assumes**: 依赖静态多视角有位姿数据、COLMAP/SfM 初始化、以及 3DGS 的 CUDA 栅格化实现；若想达到论文强调的 25×+ 压缩，还需要额外的 8-bit 量化、剪枝和 Huffman 编码；即便如此，附录显示推理显存仍大约在 **5.6–8.3 GB**，还不是移动端级别。
- **Not designed for**: 动态场景、无位姿/无 SfM 初始化场景、以及把所有属性统一交给单一神经场处理的设定。

### 可复用组件
1. **联合重要性剪枝**：同时看体积与透明度，而不是只看 opacity。  
2. **按属性类型选共享机制**：颜色用连续场，几何用字典索引。  
3. **训练期持续压缩**：不是后处理裁剪，而是边训练边保持紧凑。  
4. **测试前预计算**：把共享表示的额外开销移出在线渲染路径。

### So what
如果你关心的是 **3DGS 如何真正走向部署**，这篇论文给出的答案很实用：  
**不要只做后处理压缩，而要把“哪些高斯该存在、哪些属性该共享”直接做成训练目标。**
