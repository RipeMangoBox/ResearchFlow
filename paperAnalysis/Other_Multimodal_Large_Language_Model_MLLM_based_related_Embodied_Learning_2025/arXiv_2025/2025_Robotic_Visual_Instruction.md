# Robotic Visual Instruction

> [!abstract] **Quick Links & TL;DR**
> - **Links**: [arXiv](https://arxiv.org/abs/2505.00693), [Project](https://robotic-visual-instruction.github.io/)
> - **Summary**: 这篇工作把“手绘圆圈+箭头+颜色”的2D视觉符号变成机器人可执行的3D操作约束，用 VLM 负责语义理解、用关键点模块负责空间落地，从而显著缓解语言指令在空间精度和多步时序上的歧义。
> - **Key Performance**: VIEW-LLaVA-13B 在 8 个真实世界未见任务上平均成功率 **87.5%**；VIEW 在 3 个模拟任务上平均成功率 **76.6%**，高于 RT-1-X / Octo 语言或目标图像基线。

> [!info] **Agent Summary**
> - **task_path**: 带RoVI手绘标注的初始RGB观测 / 桌面操控任务 -> 机械臂3D动作序列
> - **bottleneck**: 人类意图里的路径、相对位置和步骤顺序很难用自然语言低歧义地传给机器人，尤其在杂乱场景、轨迹跟随和长时任务中
> - **mechanism_delta**: 把“做什么、按什么顺序做”交给VLM，把“精确从哪里到哪里”交给直接从符号提取的关键点约束，而不是让语言或端到端VLA隐式猜测全部时空信息
> - **evidence_signal**: 8个真实世界未见任务上87.5%成功率，明显高于VoxPoser和CoPa
> - **reusable_ops**: [symbol-keypoint-grounding, color-based-step-decomposition]
> - **failure_modes**: [free-form-sketch-variance, small-vlm-comprehension-drop]
> - **open_questions**: [open-model-scaling, 2d-sketch-to-3d-contact-coverage]

## Part I：问题与挑战

这篇论文真正要解决的，不是“机器人会不会抓”，而是**人如何用低负担、低歧义、可静默的方式，把时空操作意图传给机器人**。

### 1. 现有人机指令接口的真实瓶颈
- **自然语言**很方便，但它天然偏“语义级”，不擅长表达像素级/路径级空间约束。  
  例如“把柠檬移动到土豆下方并靠近一点”，对人很自然，对机器人却仍然模糊。
- **目标图像**能表达终态，但不表达过程；用户还要先想象完成后的场景。
- **完整轨迹**能表达路径，但让用户直接画出机械臂全程运动并不友好，也不直观。
- 在**图书馆、医院**这类安静场景里，语音本身也不适合作为主要交互方式。

### 2. 论文给出的输入/输出接口
- **输入**：初始观测图像 + 用户在图像上手绘的 RoVI 草图  
  - 圆圈：交互点/affordance
  - 箭头：路径与方向
  - 颜色：时间顺序
  - 数字：双臂系统的执行主体
- **输出**：机器人可执行的 3D 动作序列，而不是只给一个文本理解结果。

### 3. 边界条件
这套方法并不是通用一切机器人任务的“万能语言”，它更适合：
- **object-centric** 的桌面操作；
- 能被拆成“移动 / 旋转 / 选择抓取 / 开关抽屉柜门”等基本动作；
- 用户和机器人共享或近似共享一个**2D观测视角**；
- 任务可通过若干关键点和步骤顺序表达。

### 4. 为什么是现在
因为现在同时具备了三件事：
1. **VLM** 已经足够擅长读图、读符号、生成计划和代码；
2. **RGB-D + grasp module + motion planning** 让 2D 到 3D 的落地链路成熟；
3. 可以用 **PEFT/LoRA** 给中小型模型专门补齐这种“草图-机器人”能力。

一句话总结：**瓶颈从来不是控制器本身，而是“用户意图的可执行表示”**。

## Part II：方法与洞察

论文分成两个层次：
- **RoVI**：一种新的视觉指令语言
- **VIEW**：把这种指令翻译成机器人动作的执行工作流

### 方法拆解

#### 1. RoVI：把 3D 操作压缩成 2D 符号语言
作者把复杂操作抽象成两类视觉原语：
- **Arrow**：起点、若干 waypoint、终点
- **Circle**：交互中心点，可表示抓取点、旋转中心、按压点

再用颜色编码步骤顺序：
- 绿：第一步
- 蓝：第二步
- 粉：第三步  
更多步骤可扩展更多颜色。

这个设计的关键不是“画得像”，而是**把时空约束变成简单、稳定、可读的符号**。

#### 2. VIEW 的上层：VLM 负责语义理解与任务分解
VLM 输入：
- 手绘 RoVI
- 初始观测图像
- 系统默认 prompt

VLM 输出两类中间结果：
- **语言响应**：任务定义、子目标序列
- **代码函数**：如 `move()`、`grasp()` 这类可执行技能调用

这里的好处是：
- 人可以检查语言结果，确认机器人“理解对没对”；
- 下游控制不用从零学 SE(3) 连续动作，可复用已有技能函数。

#### 3. VIEW 的中层：关键点模块负责空间与时序约束
这是论文最关键的结构设计之一。

作者没有让模型去从场景里“猜”空间目标，而是直接：
- 按**颜色**把多步任务拆成单步；
- 从 RoVI 符号本身抽取：
  - 起点
  - waypoint
  - 终点
  - 圆心

并用一个专门训练的 **YOLOv8** 模块做符号关键点检测，而不是做开放词汇目标检测。

这意味着搜索空间从：
- “在复杂场景中找到语义相关对象与部件”
变成
- “在图上找到箭头尾、箭头头、路径点、圆圈中心”

后者显然更稳定，也更不容易被环境干扰物打断。

#### 4. VIEW 的下层：低层策略把 2D 关键点执行成 3D 动作
- 用 RGB-D 把 2D 像素关键点映射到 3D
- 用 AnyGrasp 选取接近起点的抓取位姿
- 低层控制器按关键点序列逐点执行
- 对平移和旋转分别处理

所以整条链路是：
**符号语义理解** 和 **几何轨迹落地** 被明确拆开了。

#### 5. RoVI Book：让小模型也学会这种接口
作者构建了 **15K** 的 RoVI Book：
- 基于 Open-X Embodiment 任务改写
- 每条数据包含带 RoVI 的图像、查询、回答
- 回答由 GPT-4o 生成，再经人工反馈过滤
- 对 RoVI 做 3–8 种视觉变体增强

其作用不是提出新算法，而是证明：
- 这种能力可以被**专门蒸馏**到较小 VLM；
- 为边缘部署留下可能性。

### 核心直觉

**改变了什么？**  
把机器人指令从“自然语言/终态图像”改成“显式编码路径、接触点和时间顺序的符号草图”。

**哪类瓶颈因此改变？**
1. **空间信息瓶颈**：从 object-level 模糊描述，变成 pixel-level 显式锚点；
2. **时序推理瓶颈**：从整段长时任务联合推理，变成按颜色拆分的逐步执行；
3. **感知搜索瓶颈**：从在复杂场景里做开放语义定位，变成检测简单稳定的 RoVI 符号。

**能力上发生了什么变化？**
- 轨迹跟随更可控；
- 杂乱场景里更不容易选错对象；
- 多步任务更容易对齐执行顺序；
- 人机交互从“说清楚”变成“画清楚”。

### 为什么这套设计有效
它本质上是在做一个**因果解耦**：

- **VLM** 处理高层语义：这是“什么任务、分几步”
- **关键点模块** 处理精确几何：这是“从哪里到哪里、经过哪里”
- **低层策略** 处理执行闭环：这是“如何稳定到达这些位姿”

相比让一个端到端 VLA 同时学会所有事情，这种拆法把问题难度改小了，也让错误更容易被定位。

### 战略取舍

| 设计选择 | 改变的瓶颈 | 收益 | 代价 |
|---|---|---|---|
| 手绘符号替代自然语言 | 空间表达从模糊语义变成显式几何 | 更强空间精度、静默交互 | 需要用户会在图上标注 |
| 颜色拆解多步任务 | 降低长时规划复杂度 | 多步执行更稳定 | 步骤数受颜色区分能力约束 |
| 直接检测 RoVI 符号，而非场景开放词汇检测 | 缩小感知搜索空间 | 对杂乱和干扰更鲁棒 | 依赖符号清晰可见 |
| 语言+代码中间层 | 增强可解释性与模块复用 | 便于人工核查、替换低层技能 | 管线更长，可能有误差传播 |
| RoVI Book 微调小模型 | 降低对超大闭源模型依赖 | 有边缘部署潜力 | 仍需专门数据和标注清洗 |

## Part III：证据与局限

### 关键证据信号

#### 1. 比较信号：真实世界未见任务上的显著提升
最强证据来自 8 个真实世界未见任务：
- **VIEW-LLaVA-13B (RoVI Book)**：87.5%
- **VIEW-GPT4o**：82.5%
- **VoxPoser**：43.8%
- **CoPa**：45.0%

这说明能力跃迁主要出现在：
- **杂乱环境**
- **轨迹跟随**
- **多步操作**

也就是说，RoVI 的优势不是“通用常识更强”，而是**指令接口本身把时空约束表达得更清楚**。

#### 2. 比较信号：模拟环境里优于语言/目标图像接口
在 3 个模拟任务上：
- **VIEW**：76.6%
- **RT-1-X**：20.0%
- **Octo-goal-image**：13.3%
- **Octo-language**：3.0%

论文的解释很合理：  
- 语言更偏“说目标”
- 目标图像更偏“说终态”
- **RoVI 同时说清了路径 + 终态 + 顺序**

所以它在长时和 clutter 场景更占优。

#### 3. 诊断信号：不是所有 VLM 都天然会“看懂”RoVI
- 大模型如 **GPT-4o / Gemini / Claude** 在 in-context 下具备较强 RoVI 理解能力；
- 文中测试的 **<13B 小模型** 零样本几乎不行；
- 但 **RoVI Book 微调后的 LLaVA-13B** 能把执行效果拉起来。

这说明：
- RoVI 不是“模型天然就会”的输入格式；
- 但它是**可学习、可蒸馏**的接口。

#### 4. 消融信号：关键点模块与绘制风格都很重要
- 专用 **YOLOv8 符号检测** 比开放词汇检测器更稳，mAP 达到 1.00；
- **Geometric style** 比 Loose free-form 更容易被模型理解。

这两个结果都支持同一个结论：  
**RoVI 的价值不仅在“画图”，更在“把图形约束成机器容易读的视觉语法”。**

### 1-2 个最值得记住的指标
- **87.5%**：真实世界未见任务平均成功率
- **1.00 mAP**：RoVI 关键点符号检测精度

### 局限性
- **Fails when**: 草图过于随意、颜色混淆、箭头/圆圈遮挡严重、步骤过长，或任务需要细致的3D接触几何而不是2D路径提示时，RoVI理解与执行都会下降。
- **Assumes**: 需要共享观测图像上的标注界面、RGB-D与相机标定、AnyGrasp等抓取模块；最强理解能力依赖闭源GPT-4o，而小模型版本依赖RoVI Book这类专门数据与人工过滤；评测规模也主要是11个任务、每任务10次试验。
- **Not designed for**: 移动导航、全身操作、复杂双臂协同、强形变物体处理、需要连续力控制/装配精度的任务，也不是一个纯端到端学习框架。

### 可复用组件
- **RoVI 视觉指令语法**：圆圈/箭头/颜色/编号
- **按颜色分步的时序拆解**：适合长时任务
- **symbol-level keypoint grounding**：从符号而不是场景语义取约束
- **语言+代码中间接口**：方便接入不同低层策略
- **RoVI Book 数据构造范式**：可继续扩到更多自由手绘风格

### 一句话结论
这篇工作的核心贡献，不只是提出一种“能画给机器人看”的交互方式，而是证明了：**当人类意图被压缩成显式、可定位、可分步的视觉符号后，机器人在杂乱、多步、轨迹敏感任务上的泛化会明显提升。**
