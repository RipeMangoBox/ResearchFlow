你要做的不是一个“论文仓库”，而是一个“研究操作系统”。
系统要把人保留在两个位置：第一是定义现在真正的瓶颈，第二是判断哪些直觉值得押注；其余流程——收集、初筛、结构化分析、对比、阅读排序、周期总结、idea 拼装、复盘记忆——都应该尽量由系统一步到位地完成。

还有一个很关键的边界：不要再把 Claude Code 或某个单一 agent 当成系统本体。系统本体应该是“对象存储 + 数据库 + 异步任务 + Web 前端 + MCP 适配层”；Claude Code、Codex、未来别的 agent 都只是接入这个后端的“专家客户端”。之所以这样定，是因为 PostgreSQL 本身已经有 jsonb 索引和全文检索，pgvector 默认支持 exact nearest-neighbor search，并可加 HNSW / IVFFlat；等以后 hybrid query、payload-heavy filtering、named vectors 更重时，再把高级检索拆到 Qdrant 会更自然。

我这里按 2000 元/年基础设施预算，不含模型 token 预算 来设计。这个预算下，真正贵的是常驻服务器，不是对象存储。腾讯云轻量 2核4GB/100GB 的官方页是约 1020 元/年，4核8GB 已经到 2499 元/年；腾讯 COS 标准存储大约 0.118 元/GB/月，按这个价格算 200GB 一年约 283 元、500GB 一年约 708 元。阿里 OSS 的中国内地标准存储大约 0.12 元/GB/月，200GB 一年约 288 元、500GB 一年约 720 元，请求费也都很低。也就是说，在你的预算里，1 台 2C4G 常驻节点 + 托管对象存储是甜点位，4C8G 常驻节点反而会把预算直接吃穿。

一、系统应该怎么重构

我建议把现有项目改成五层：

原始证据层
PDF、补充材料、项目页 HTML、代码仓库快照、图片、表格截图，全都进对象存储。
这里存“原件”，不做推理。
结构化事实层
用 Postgres 做系统事实源。
这里存 paper 元数据、抽取结果、方法分类、证据片段、阅读计划、项目瓶颈、总结、用户反馈、任务状态、模型版本、执行记忆。
检索层
第一阶段直接用 Postgres + pgvector。
关键词检索、结构化过滤、语义检索都先放一起，简单、便宜、透明。
只有当你后面真的出现“大量 payload 过滤 + hybrid 检索 + named vectors + 并发更高”时，再把高级检索拆到 Qdrant。
工作流层
API 服务 + 异步 worker + 定时任务。
这层负责收集、去重、筛选、分析、重分析、生成 digest、导出 markdown。
交互层
普通用户走 Web 前端。
高级用户、维护者、研究工程师，才走 Claude Code / Codex / IDE agent。

你现在的 paperAnalysis/、paperCollection/、paperIDEAs/、.claude/skills/ 这些都不该删。
但它们要降级成 导出层 / 兼容层，而不是 source of truth。
也就是说：数据库是主，Markdown 是导出物。
这样你仍然保留可审计性、Obsidian 兼容性、CLI 可读性，同时摆脱 CSV/文件夹在并发、增量更新、版本追踪上的脆弱性。

二、论文知识库怎么存、怎么同步，才兼顾性价比、速度、性能、token 消耗

课题组场景最稳的方案是：

Zotero 继续保留，作为“人类可见的收集入口”
后端定时从 Zotero 同步条目
用 Crossref 补全 DOI、license、ORCID、abstract、funding、更新信息
PDF 和补充材料进对象存储
Postgres 存结构化元数据和分析状态

Zotero 官方同步覆盖 item metadata、notes 和 full-text content；Crossref REST API 也能补全大量标准学术元数据。这个组合很适合课题组，因为 Zotero 很适合“人手工收集与分组”，而系统后端更适合“机器批量补全与分析”。

真正决定 token 消耗的，不是存储，而是你是不是反复让模型读整篇论文。
所以知识库一定要做成分层分析，而不是“论文来一篇，模型全文读一遍”：

第一层，metadata-only。
只用题目、摘要、venue、年份、代码/数据开源状态、作者团队、来源链接，先做候选管理和初筛，不上大 token。

第二层，local parse。
先用本地 PDF 解析拿全文、章节、公式块、表格附近文字、图注。这个阶段尽量不用 LLM。

第三层，skim card。
只让模型看 title / abstract / intro / method overview / conclusion，生成轻量卡片：这篇想解决什么、改了哪个槽位、是不是插件修补、值不值得深读。

第四层，deep report。
只有进入高优先队列的论文，才做全文深度分析、证据对齐、跨论文对比、迁移性评估。

这样做的本质，是把“是否值得花 token”变成一个明确的决策点。
系统不应该对所有论文一视同仁。

我建议对象存储按下面的桶来分：

papers/raw-pdf/
papers/raw-html/
papers/extracted-text/
papers/figures/
papers/code-snapshots/
reports/skim/
reports/deep/
exports/markdown/

数据库里至少要有这些主表：

papers
paper_assets
paper_versions
paper_analyses
method_deltas
evidence_units
transfer_atoms
project_bottlenecks
search_sessions
reading_plans
digests
execution_memories
jobs
model_runs
user_feedback

三、论文提取后的“核心 idea”到底怎么抽象

你说得很对：
只存 PDF 太冗余；只存公式太抽象；只存代码太细碎。
所以真正该存的，不是“论文全文”，而是研究知识原子。

我建议一篇论文最后至少拆成 6 类原子：

问题原子
它到底解决哪个瓶颈，不解决什么。
机制原子
它改的是哪个槽位，用了什么机制，而不是“讲了什么故事”。
证据原子
有哪些实验、ablation、对比，哪些证据真的支撑因果，哪些只是相关性。
边界原子
它在哪些数据条件、训练范式、任务长度、评价协议下才成立。
迁移原子
这个 insight 可不可以搬到别的领域；前提条件是什么；哪里会失效。
复现原子
数据、训练代码、推理代码、权重、环境、踩坑记录、失败修复。

这里最重要的是：每篇论文都必须有一个“canonical delta card”。
也就是把它和领域内标准范式对齐，明确写清：

领域标准流水线是什么
这篇改了哪几个槽位
哪些槽位完全没改
改动是“插件修补”还是“结构性改动”
核心增益更像来自 objective、credit assignment、planner、memory、search，还是只是权重/后处理/小技巧

例如 RL 里，标准范式卡片就要固定有：

policy
value / baseline
reward / advantage
update rule
rollout / exploration
planner / memory（如果有）

VLM 里也一样，要有：

perception encoder
projector / connector
language core
alignment objective
data mixture
inference strategy / planner

这样模型就不会再顺着论文自己讲的故事走，而是先对照“标准范式”，再讲它到底动了哪里。
这一步是解决你说的“LLM 被论文叙事骗了”的关键。

四、过滤、分类、排序、阅读推荐应该怎么做

这里一定不要只做一个总分。
至少拆成四个分：

keep_score
决定这篇要不要进库、进哪个池子。
analysis_priority
决定先分析谁。
structurality_score
区分它是插件 patch，还是结构性改进。
extensionability_score
判断它有没有可扩展、可迁移、可组合的潜力。

你的初筛规则我建议直接做成分层，不要绕：

A 层：开数据优先
B 层：开训练代码其次
C 层：不开源但已中稿，进入结构化存档池
D 层：没中稿 / 仅预印本 / 不确定质量，进入观察池，再看团队、写作质量、时间衰减

然后在每层内部，再按下面这几个因素排序：

团队可信度
任务相关性
机制结构性
复现资产完整度
新鲜度
证据强度

时间衰减不要一刀切。
对“没中稿、且没开源”的 paper，可以衰减更快；
对“经典、结构性、被后续大量继承”的 paper，衰减要慢很多。

方法分类我建议不要只靠标签，也不要只靠树状 taxonomy。
应该同时有三层：

第一层，硬分类
比如 domain、task、changed_slot、mechanism_family、supervision_type、inference_pattern。

第二层，软标签
比如“插件型”“reward patch”“group reward”“agentic planner”“self-correction”“think with image”“credit decomposition”这些社区说法和你自己的工作语汇。

第三层，搜索分支树
这个最重要。因为很多时候用户说不清自己真正要什么，LLM 直接按 query 推荐会很短视。
所以系统要显式存：

symptom_query：用户最初怎么描述问题
latent_need：系统推测他真正想要什么
candidate_bottlenecks
rejected_solution_patterns
search_branches

你举的 RL 例子就非常典型：

最初症状：优势会消失。
第一轮检索确实会进很多 RL 论文，但发现多数只是 plugin 式 patch，比如改 advantage weighting。
接着系统应该把“latent need”重写成：
“需要更可扩展的 credit / reward 管理，或者更 agentic 的扩展式方案”。
于是才会自动开出新的 branch：grouped reward、planner-style、Think-with-Image 这一类 agentic 路线。

也就是说，系统不能只回答“有什么相关论文”，还要回答“为什么要换一条检索支路”。

阅读推荐也不能只是按相似度排。
我建议默认顺序是：

先读 1～2 篇 canonical baseline
再读 2～3 篇结构性改进
再看最新的强团队跟进
最后再看插件型 patch 和负例

这样用户不会一上来就被一堆 patch paper 淹没。

五、并发 agent 和增量 agent 应该怎么设计

并发 agent 不是把一件事拆成十个模型一起跑，而是把不同稳定职责分开：

Collect Agent：收集与去重
Triage Agent：开源/中稿/团队/新鲜度初筛
Skim Agent：轻量卡片
Deep Report Agent：全文深度报告
Taxonomy Agent：分类与槽位映射
Evidence Auditor：证据核查，防叙事欺骗
Reading Planner：阅读顺序与详略推荐
Bottleneck Reframer：把“模糊需求”重写成“瓶颈定义”
Idea Composer：跨领域拼接机制
Reviewer Agent：审稿人视角找漏洞
Digest Agent：日/周/月总结
Execution Memory Agent：环境与踩坑记忆沉淀

在 2C4G 的机器上，不要想着本地跑大模型。
它只适合做编排和轻量处理。
我的建议配置是：

1 个 API 服务
1 个 scheduler
1 个 parser worker
2 个 LLM worker 槽位
1 个 embedding 批处理槽位
Postgres + Redis 同机

也就是：
实时交互以读为主，重分析一律排队。
普通搜索、看报告、看阅读清单都应该秒开；
深度报告、跨论文对比、月度总结这种重活异步跑。

增量 agent 的核心不是“记住上次聊天”，而是“记住系统为什么这么判断”。
所以每次分析都要带版本：

model_provider
model_name
prompt_version
schema_version
evidence_spans
confidence
generated_at

这样后面模型变强、taxonomy 更新、出现新反例时，系统就能只重跑该重跑的部分，而不是把整库全洗一遍。

触发重分析的原因至少要有：

模型升级
taxonomy 变化
新论文与旧结论冲突
用户纠错
某领域最近一波论文出现新共识
某篇高优先 paper 的旧分析置信度过低

你提到“第一次配环境很乱，第二次不应该再乱”，这部分我建议单独做 execution_memory：

环境指纹：repo + OS + Python + CUDA + 依赖摘要
失败命令
修复动作
是否验证成功
适用条件
失效条件

这样 agent 第二次不是凭记忆瞎试，而是先检索执行记忆。
这比单纯给模型更多上下文更稳。

另外，自进化不能只靠 agent 自我感觉。
最好保留一套固定 eval 集：比如 50 篇标杆论文、10 个典型搜索问题、10 个典型模糊需求重构场景。
每次模型、prompt、taxonomy 改了，都过这套集，防止“能力看起来增强了，但结构性判断变差”。

六、日总结、周总结、月总结怎么设计

这个模块不要当成“额外写三份文章”，而要当成同一套知识状态的三个视图。

日总结
适合解决“今天值得看什么”。

应该包含：

新入库高优先论文
优先级变化最大的论文
今天最值得深读的 3 篇
今天新增的瓶颈假设 / 被否决的支路
需要人确认的 1～3 个关键判断
今晚适合后台继续跑的任务

周总结
适合解决“这周方向有没有发生偏移”。

应该包含：

本周哪些主题升温
哪些只是 patch paper 堆积
哪些结构性路线开始成形
阅读分支树怎么变了
哪些旧判断被推翻了
下周最值得追的 3 条线

月总结
适合解决“这个月要不要改研究策略”。

应该包含：

领域地图更新
当前最可信的瓶颈排序
结构性路线 vs 插件路线的胜负态势
哪些 idea 值得立项
哪些 paper 可以归档不再追
哪些旧论文要用更强模型重分析

存储上不要重复存三份正文。
应该存：

digest_scope
period_type（day/week/month）
source_paper_ids
source_bottleneck_ids
source_search_session_ids
rendered_text
render_version

这样以后你可以重渲染，不用重算全链路。

七、普通用户到底需要看到什么

普通用户真正要的信息，不是“库里有多少论文”，而是下面这几项：

这篇值不值得看
为什么值不值得看
它改了哪个关键槽位
它是结构性改进还是 patch
它的证据到底强不强
和我当前瓶颈有什么关系
推荐阅读顺序是什么
应该看 30 秒版、5 分钟版，还是 20 分钟深读版
还有哪些方向已经被系统判定为“暂时不用浪费时间”

所以前端至少应该有 5 个入口：

搜论文
快速读
对比论文
找 idea
看日/周/月总结

普通用户的输入尽量缩到最少，只问两类：

我现在卡在哪个现象 / 瓶颈
哪些约束不能碰（数据、算力、延迟、训练方式、任务边界）

而不是逼用户先说清领域 taxonomy。

八、用户需不需要跑 Claude Code

普通用户不需要。

Claude Code 本质上是一个 terminal / IDE / desktop / browser 里的编码 agent。官方文档里它是一个会读代码、改文件、跑命令、接 development tools 的 coding tool；它既能在终端里跑，也能在浏览器里无本地启动长任务和并行任务，但桌面/网页 Code 入口需要付费订阅。这个形态很适合开发者、维护者和高级分析师，不适合作为 0 门槛产品的唯一入口。

所以最合理的角色分工是：

普通研究生/老师：只用 Web
高级用户：可选 Claude Code / Codex 做深度交互
维护者：用 Claude Code / Codex 开发技能、修规则、审结果
后端服务：自己跑 API + worker，不依赖人工打开 Claude Code 会话

九、怎么适配 Claude Code，才真正发挥它的价值

Claude Code 这条线我会把它放在“专家操作台 + 开发工作台”上，而不是后端本体上。
理由很直接：官方现在已经有 MCP、skills、subagents、hooks、Agent SDK 这整套能力，已经足够强，非常适合做专家工作流；但系统状态、任务队列、对象存储、用户会话这些持久东西，不该挂在 CLI 会话里。Claude Agent SDK 甚至明确提供了和 Claude Code 相同的 tools、agent loop 和 context management，适合直接在 Python / TypeScript 里做服务端编排。

Claude 侧我建议你这样接：

后端暴露一个 ResearchFlow MCP server
工具只给这几个：
search_papers
get_paper_card
get_deep_report
compare_papers
upsert_project_bottleneck
create_reading_plan
enqueue_analysis
get_digest
record_feedback
query_execution_memory
在项目里放几类 skill
放在 .claude/skills/，因为 Claude Code 原生就是 SKILL.md 目录结构，项目级 skill 也天然适合团队协作。Claude 会自动发现这些 skill，改动还能在当前 session 里生效。

我建议至少有这几个：

research-triage
deep-paper-report
bottleneck-reframer
cross-domain-transfer
weekly-research-review
reviewer-stress-test

其中 deep-paper-report 和 cross-domain-transfer 这种可以加 context: fork，让 skill 直接跑在隔离 subagent 里，避免主对话被污染。Claude Code 官方也支持 skill 这样跑。

再加两个专用 subagent
放在 .claude/agents/：
evidence-auditor
taxonomy-reviewer

Claude Code 官方支持项目级和用户级 subagent，内建也有自动 delegation 的 subagents。你这里完全可以把“证据核查”和“分类对齐”做成专门代理。

用 hooks 做确定性动作
比如：
每次 deep report 结束，自动写回数据库
每次生成 digest，自动导出 markdown
每次更新 taxonomy，自动触发校验
每次 agent 要写某些受保护字段，先跑校验脚本

Claude 的 hooks 是确定性的 shell command，不是“让模型自己决定要不要做”。这特别适合你这种高可靠流程。

MCP 配置可以直接共享
Claude Code 官方支持把 MCP server 设成项目作用域，写到 .mcp.json 里，方便团队共享。命令行也支持直接添加 HTTP MCP server。

你可以把它想成这样：

Claude Code = 超强人工驾驶台
ResearchFlow backend = 真正的飞机

十、怎么适配 Codex，以及以后别的 agent 框架

Codex 这边现在也已经非常适合接这套后端。
官方文档里，Codex CLI 是本地 terminal coding agent；配置走 ~/.codex/config.toml 或项目级 .codex/config.toml；它会在开始工作前自动读 AGENTS.md；它也有 skills、subagents、MCP，而且 Codex 本身还能跑成一个 MCP server 供别的 agent 消费。默认网络还是关的，配合 sandbox 和 approval policy，安全边界也比较清楚。

所以 Codex 适配策略很简单：

继续保留 AGENTS.md，但只写“地图”，别写成百科
OpenAI 官方最佳实践也强调了这点：AGENTS.md 适合放 repo layout、命令、约束、done definition；太长就拆出去引用别的文档。
项目里放 .codex/config.toml
把同一个 ResearchFlow MCP server 配进去。
类似这样：
[mcp_servers.researchflow]
url = "https://research.example.com/mcp"
startup_timeout_sec = 20
tool_timeout_sec = 120
enabled_tools = [
  "search_papers",
  "get_paper_card",
  "get_deep_report",
  "compare_papers",
  "create_reading_plan",
  "get_digest"
]

Codex 官方文档就是用 [mcp_servers.<name>] 这种表来配 HTTP MCP server，CLI 和 IDE 共享同一套配置。

共享一套中立 skill 源
Codex 的 skills 也是 SKILL.md + 可选脚本/资源 的目录结构，而且官方说它建立在 open agent skills standard 上。
所以你完全可以把 skill 的“中立源文件”放一份，然后生成 Claude 版和 Codex 版包装层。
后面要接别的 agent，就坚持两个标准
一个是 MCP 工具边界，另一个是 AGENTS.md / SKILL.md 这类开放文档边界。
这样你不会被某一家锁死。
更进一步时，可以做“agent 套 agent”
因为 Codex 官方支持把 Codex 自己作为 MCP server 暴露出来，供其他 agent 调用。
这意味着后面你可以做这样的编排：
Claude 负责 research reasoning，Codex 负责重构解析器或实现后端工具；或者反过来让 Codex 调 Claude 的研究工具。

而且 OpenAI 这边现在也明确支持 remote MCP servers 和 connectors，不只是 Codex CLI，连 ChatGPT Apps / API integrations 也能接进来。也就是说，你后面要做成“ChatGPT 里直接连 ResearchFlow”的形态，是有官方通路的。

十一、2000 元预算下，我给你的具体推荐配置

我会直接给一个能落地的版本。

基础假设：
这是“年化基础设施预算”，不含模型 token。
如果 2000 元连 token 也要一起算，那方案就得进一步瘦身，只能做很轻的分析和更小的样本池。

推荐配置：

云服务器：1 台 2C4G，100GB SSD，Ubuntu 24.04
对象存储：COS 或 OSS，先开 200GB
数据库：Postgres 16 + pgvector，自托管在这台机器上
队列：Redis，自托管
API：FastAPI
前端：Next.js
反向代理：Caddy 或 Nginx
部署：Docker Compose

预算上，2C4G + 200GB COS + 约 80 元域名 + 100 元备份预留，大约 1483 元/年；
换成 500GB COS 也大约 1908 元/年。
换成阿里 OSS 也差不多，在 1488～1920 元/年的区间。

服务器内部分配我建议这样：

Postgres：1.2GB
Redis：256MB
API：400MB
Worker：1GB
反向代理 + 系统：1GB 左右
剩余留作 buffer

并发建议：

实时检索：走数据库，尽量缓存
重分析：同时最多 2 个
PDF 解析：同时 1 个
embedding 批处理：低峰跑

也就是说，它适合：

3～5 个活跃用户看结果
1～2 个后台重分析任务持续跑

不适合：

本地大模型推理
大量 OCR
大量高并发全文重分析

所以模型一定要走外部 API，服务器只做 orchestration、解析、缓存、检索、导出。

十二、真正该先做的三步

第一步，先把数据模型和导出链路立住。
也就是把 analysis_log.csv、frontmatter、paper reports 迁进数据库，同时保留原来的 Markdown 导出，保证老工作流不断。

第二步，把“按论文存东西”改成“按瓶颈和机制存东西”。
先把 method_delta、evidence_unit、project_bottleneck、search_session 这四类核心实体立起来。
没有这一步，后面 idea 模块一定短视。

第三步，把同一个后端同时接到 Web、Claude Code、Codex。
这样普通用户不碰 CLI，高级用户又能用最强的 agent 工作台，未来迁移到别的框架时也不会推倒重来。

我现在最强烈的建议只有一句：
先把“数据库事实源 + MCP 工具边界 + canonical delta card”这三个基础设施做出来，别再继续把 CSV/Markdown/单 agent prompt 当主系统。
